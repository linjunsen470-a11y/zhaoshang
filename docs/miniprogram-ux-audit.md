# 校园商铺招商小程序 — 用户体验审计报告

> 审计日期：2026-06-21  
> 实施更新：2026-06-22  
> 审计范围：`mp/` 微信小程序客户端  
> 审计视角：以意向商户的真实决策路径为核心（发现铺位 → 比对信息 → 提交咨询 → 等待跟进 → 成交/转让/设备撮合）

**实施状态摘要：** P0 中除 3.3（生产 API）、3.6（收藏服务端同步，已改为本机提示）外均已修复；P1 大部分已落地；P2 仍待规划。代码提交见 `cc7ae59`。

## 一、用户画像与核心需求

| 用户类型 | 核心诉求 | 当前产品对应能力 |
|---------|---------|----------------|
| 想开店的新手 | 快速找到匹配预算/业态的档口，了解费用与客群 | 首页推荐 + 多维筛选 + 详情页 |
| 有经验的经营者 | 精准筛选学校/区域，收藏对比，高效联系顾问 | 收藏、筛选、一键拨号 |
| 转让方 | 委托平台撮合接手方，保护隐私 | 转让表单 + 顾问对接 |
| 设备买卖双方 | 发布/浏览设备供需 | 设备列表 + 发布表单 |
| 提交咨询后 | 知道进度、能改需求、能删错单 | 咨询记录 + 状态时间线 |

整体信息架构（首页 / 找铺 / 我的）清晰，**主流程骨架是完整的**。

## 二、做得好的地方（应保留）

### 2.1 发现与筛选路径顺畅

- 首页提供搜索、金刚区（找铺主入口 + 转让/餐饮设备次入口）、业态快捷入口、推荐横滑、最新列表。
- 找铺页支持区域、类型、预算、业态、状态五维筛选，有空状态与重置引导。

### 2.2 详情页信息密度合理

- 轮播图、状态标签、基础参数、转让信息、适合/不适合业态、亮点、客群、合作方式、看铺安排分层展示。
- 底部固定操作栏：收藏、分享、联系顾问、我要咨询；已满/已下架时禁用咨询按钮，符合用户预期。

### 2.3 线索闭环较完整

- 提交后跳转成功页，引导查看记录或回首页。
- 「我的咨询记录」支持三段处理状态、编辑、删除和图片预览；后续沟通明确以微信或电话为准。
- 表单自动填充用户称呼/手机号，减少重复输入。

### 2.4 工程层面的体验保障

- 图片上传前压缩、最多 6 张、上传中禁用提交。
- 401 自动刷新 token 重试。
- 设备列表隐藏联系方式，通过顾问对接，保护隐私。

## 三、严重问题（P0 — 直接影响信任与可用性）

### 3.1 「开店测算」名不副实 — 预期落差最大 ✅ 已修复

**原状：** 首页入口写的是「轻量预算评估」，实际只是打开咨询表单并预填一句备注。

**实施（2026-06-22）：** 已移除该入口，避免承诺与实现不一致。

```javascript
// mp/pages/index/index.js
} else if (action === 'budget') {
  wx.navigateTo({ url: '/pages/apply/apply?leadType=leasing&from=budget' });
}
```

**用户影响：** 用户会以为有计算器或测算结果，实际仍是「填表等顾问回电」。这是典型的体验落差，容易引发差评与流失。

**建议：** 要么做简易测算（租金 + 设备 + 人工 ≈ 启动资金区间），要么改名为「预算咨询」并改描述。

---

### 3.2 首页「推荐项目」快捷入口逻辑错误 ✅ 已修复

**原状：**

```javascript
// mp/pages/index/index.js
{ id: 'cat8', name: '推荐项目', icon: '🔥', type: 'status', val: 'online' },
```

点击后筛选的是「开放中」而非 `isRecommended`。用户点「推荐」却看到全部在线项目，**入口语义与结果不一致**。

**建议：** 改为 `{ type: 'recommended', val: true }`，与「查看全部推荐」行为对齐。

---

### 3.3 生产环境 API 仍指向本地 ⏸ 暂缓

**现状：**

```javascript
// mp/config.js
API_URL: 'http://127.0.0.1:3000/api',
```

真机/正式版用户无法访问。失败时提示：

```javascript
// mp/pages/list/list.js
title: '加载失败，请先执行 pnpm backend:dev',
```

这是开发者文案，普通用户无法理解，且会认定产品「坏了」。

**建议：** 按环境切换 API 地址；用户侧统一为「网络异常，请稍后重试」+ 重试按钮。

---

### 3.4 咨询记录展示原始英文类型码 ✅ 已修复

**原状：**

```xml
<!-- mp/pages/leads/leads.wxml -->
<text class="spec-lbl">需求类型：</text>
<text class="spec-val">{{item.leadType}}</text>
```

用户会看到 `leasing`、`equipment_sell` 等，而非「找铺咨询」「设备出售」。专业感与可读性都受损。

**建议：** 增加 `leadType` 中文映射表，展示层统一本地化。

---

### 3.5 隐私合规缺口 ✅ 已修复

**原状：**

- 有隐私政策页，但**首次提交表单前无勾选同意**。
- 收集手机号、openid、图片等敏感信息，未在表单层明示授权。
- 微信审核对个人信息收集通常要求「告知 + 同意」。

**建议：** 首次提交前弹窗或表单底部增加「已阅读并同意《隐私政策》」勾选；未勾选不可提交。

---

### 3.6 收藏仅存本地，换机即失 ⏸ 部分缓解

**现状：** 收藏仍写入 `wx.setStorageSync('favorites')`，未与账号同步。

**用户影响：** 用户换手机、清缓存、重装后收藏消失；项目下架后收藏项也无法清理提示。对「比对多个铺位」的高意向用户，这是核心功能可靠性问题。

**实施（2026-06-22）：** 「我的」页与收藏列表已增加「收藏仅保存在本机」提示。服务端同步仍待后续。

## 四、重要体验问题（P1 — 影响效率与满意度）

### 4.1 列表页频繁无意义刷新 ✅ 已修复

**原状：**

```javascript
// mp/pages/list/list.js
onShow() {
  // ...
  this.loadProjects();
},
```

从详情返回、切换 Tab 回来都会重新请求并 `loading: true`，造成闪烁和流量浪费。

**建议：** 仅在筛选变化、下拉刷新、从首页带 `pendingListFilter` 时刷新；从详情返回使用缓存。

---

### 4.2 首页无加载态，易出现「空白闪一下」 ✅ 已修复

**原状：** `index.js` 有 `loading: true`，但 `index.wxml` 无骨架屏/加载指示。

**用户影响：** 弱网下首页先空白再突然出现内容，首屏体验不稳定。

**建议：** 增加骨架屏或加载指示器。

---

### 4.3 下拉刷新配置不完整 ✅ 已修复

**原状：** 仅 `leads`、`equipment-list` 开启了 `enablePullDownRefresh`；首页、找铺列表虽写了 `onPullDownRefresh`，但 json 未开启，用户下拉无效。

**建议：** 在 `index.json`、`list.json` 中开启 `enablePullDownRefresh`。

---

### 4.4 筛选能力不完整 ✅ 已修复

**原状：** `list.js` 有 `selectedOpportunityType`（招商 vs 转让），但 UI 无对应筛选项。

**用户影响：** 用户无法在一处区分「招商项目」和「转让项目」，只能靠标题或进详情判断。

**建议：** 增加「招商/转让」筛选 Tab 或下拉选项。

---

### 4.5 搜索清除按钮行为过激 ✅ 已修复

**原状：** 找铺页搜索框的「×」绑定 `onResetFilters`，会清空**全部筛选条件**，而不只是关键词。

**用户影响：** 用户只想清搜索词时会被意外重置，违背操作预期。

**建议：** 清除按钮仅清空 `keyword`，保留其他筛选条件。

---

### 4.6 用户身份体系薄弱 ⏸ 部分缓解

**现状：**

- ~~头像固定为 Unsplash 外链~~ 已改为本地默认头像；仍无法自定义上传。
- 无微信头像/昵称授权。
- 称呼/手机号靠手动编辑或首次提交反写，新用户看到「未设置称呼」缺乏归属感。

**建议：** 接入微信昵称/头像授权（或至少提供本地上传头像），降低身份建立摩擦。

---

### 4.7 拨号失败显示「模拟拨打」 ✅ 已修复

**原状：**

```javascript
// mp/app.js
fail: () => {
  wx.showToast({
    title: `模拟拨打：${phone}`,
```

正式环境若 `makePhoneCall` 失败，用户看到「模拟拨打」会困惑，损害平台可信度。

**建议：** 改为「拨打失败，请手动拨打 xxx」或提供复制号码入口。

---

### 4.8 咨询具体项目时上下文不够醒目 ✅ 已修复

**原状：** 从详情进咨询表单，项目名称只出现在描述小字中：

```xml
<!-- mp/pages/apply/apply.wxml -->
<view class="form-desc">{{projectTitle || '填写你的经营品类...'}}</view>
```

**用户影响：** 用户可能不确定「是否在咨询这个铺位」，增加误提交与焦虑。

**建议：** 在表单顶部固定展示项目卡片（图 + 标题 + 费用）。

---

### 4.9 成功页承诺「4 小时内联系」无依据 ✅ 已修复

**原状：**

```xml
<!-- mp/pages/success/success.wxml -->
我们将在 4 小时内联系你。
```

**用户影响：** 若 SLA 无运营保障，超时后用户信任崩塌。

**建议：** 改为「工作日优先联系」或展示顾问工作时间。

---

### 4.10 错误恢复能力弱 ✅ 已修复（首页/找铺）

**原状：** 网络失败、上传失败、加载失败多为 Toast/Modal 一句带过，无「重试」按钮。

**建议：** 空状态/错误态增加「点击重试」按钮。

## 五、中等优化项（P2 — 提升专业感与留存）

| 领域 | 现状 | 用户价值 |
|------|------|---------|
| 地理信息 | 仅文字地址 + 复制 | 校园铺位强依赖位置感知，缺地图/导航 |
| 对比决策 | 仅收藏 | 高意向用户常比 3–5 个项目，缺对比视图 |
| 消息触达 | 无订阅消息 | 状态变更、新匹配项目无法主动通知 |
| 关联推荐 | 详情页无「相似铺位」 | 错失二次转化 |
| 视觉 | 大量 emoji 作图标 | 偏 demo 感，商业产品宜用统一图标库 |
| 区域数据 | 区县写死广州 | 扩展其他城市需改代码 |
| 图片性能 | 列表图未 lazy-load | 长列表滑动可能卡顿 |
| 样式维护 | leads/list 大量内联 style | 不影响用户，但长期迭代成本高 |

## 九、总结

这个小程序**已经覆盖了校园招商场景的主链路**：找铺、看详情、咨询、转让、设备、跟进记录，信息架构和详情页质量都不错，说明产品思路贴近真实商户需求。

2026-06-22 实施后，原先最伤信任的 P0 问题（测算入口、推荐筛选、隐私勾选、英文类型码、模拟拨打）已修复；P1 加载与筛选效率项也基本落地。

**剩余重点：**

- 生产环境 `API_URL` 与用户侧统一错误文案（3.3）
- 收藏服务端同步（3.6，当前仅本机提示）
- P2 差异化能力（地图、订阅消息、相似推荐等）

完成上述项后，产品可从「能用的 MVP」进一步迈向「商户愿意推荐的平台」。
